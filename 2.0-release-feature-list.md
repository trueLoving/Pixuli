# Pixuli 2.0 发布版本功能清单

> 这份清单是基于当前重构计划与产品路线图整理的“2.0 版本建议发布内容”，用于说明 2.0 应该具备的核心能力与交付形态。

## 1. 2.0 版本定位

Pixuli
2.0 应该是一款“真正可稳定使用的跨端图片管理产品”，以本地工作区为核心，兼顾远端存储、插件扩展与生产力操作体验。

它的目标不是继续堆砌功能，而是提供一套统一、稳定、可长期使用的基本产品形态。

---

## 2. 核心功能清单

### 2.1 跨端统一体验

- 支持 Web、Desktop、Mobile 三端使用
- 三端共享统一的核心操作流程
- 统一的图片浏览、管理、标签、描述与批处理体验
- 统一的工作区与存储模型，避免三端体验割裂

### 2.2 本地工作区为核心

- 支持用户指定本地工作目录
- 以本地工作区作为主要管理入口
- 支持本地文件夹级别的图片组织与浏览
- 支持可选的远端同步能力，而非强依赖服务端

### 2.3 存储与同步能力

- 支持通过插件机制接入主流存储提供方，如 GitHub / Gitee
- 提供稳定的图片上传、查看、管理流程
- 支持通过本地工作区与远端存储协同使用
- 具备清晰的存储来源管理与切换能力

### 2.4 图片管理基础能力

- 图片浏览与列表展示
- 图片筛选与搜索
- 图片分类与组织
- 图片标签管理
- 图片描述与元信息管理
- 基础的图片预览与查看体验

### 2.5 图片生产力操作

- 图片批处理能力
- 批量标签/描述调整
- 批量移动/整理/管理操作
- 常用操作入口清晰、流程简化

### 2.6 内容回收与安全管理

- 支持回收站机制
- 可恢复已删除内容
- 避免误操作导致内容不可找回

### 2.7 Desktop 平台能力补齐

- 离线浏览能力
- 上传队列能力
- 自动更新能力
- 系统托盘能力
- 更适合长期使用的桌面端体验

### 2.8 插件与扩展能力

- 建立统一的插件体系模型
- 官方存储 provider 作为内置插件运行
- 插件具备独立生命周期、错误隔离与配置能力
- 后续第三方扩展具备可演进的基础

---

## 3. 2.0 版本的“必须达成”能力

以下能力建议作为 2.0 的核心交付边界：

1. 三端核心流畅使用
2. 本地工作区作为主工作流
3. 可稳定接入并管理存储插件
4. 完整的基础图片管理体验
5. 具备批处理与回收站等生产力能力
6. Desktop 端体验可用且稳定
7. 测试与发布流程可验证、可回归

---

## 4. 2.0 版本的质量要求

2.0 不仅要“有功能”，还要满足以下工程质量要求：

- 具备冒烟测试与集成测试体系
- CI 能稳定执行关键验证
- 关键路径可回归验证
- 用户常见操作无需依赖手工排查
- 发布版本具备可维护性与可扩展性

---

## 5. 2.0 版本的发布形态（总结）

2.0 的发布形态可以概括为：

- 一个真正可用的跨端图片管理产品
- 以本地工作区为中心的内容管理方式
- 具备稳定的存储插件与同步能力
- 拥有较完整的基础管理与批处理能力
- 具备更成熟的桌面端与工程质量保障

它的目标是让用户能够“稳定地使用 Pixuli 完成日常图片管理任务”，而不是停留在概念验证或试用阶段。
